Design direction for intelligent components
- Goal: evolve from graphical blocks to intelligent project components.
- A component instance should have an invisible unique `componentId`.
- The visible electrical tag should be derived from two editable properties:
  - `tagPrefix`, for example K, S, QF, M or X.
  - `tagNumber`, for example 1, 2 or 3.
- The final visible tag is composed as `tagPrefix + tagNumber`, for example K1, K2, S1 or QF1.
- During component creation, the user enters only the prefix. The app generates the next serial number for that prefix.
- The visible tag should not be an ordinary loose text shape.
- It should be a label linked to a component instance.
- If the user changes a component prefix or number, all linked visible labels should be able to update later.
- Components should support a relationship field such as `partOf`.
- Example:
  - A contactor coil is component K1.
  - A normally-open auxiliary contact may also show K1.
  - The auxiliary contact can have `partOf` pointing to the main K1 component.
- This relationship prepares the app for future cross-references.
- Future cross-references should be able to show where related component parts are located in the project, such as page and marker address.

Core concept distinction
- `ComponentDefinition`
  - Reusable graphical/template data.
  - Similar to the current Component object.
  - Can live in App Library or Project Components.
- `ComponentInstance`
  - A real placed component in the project.
  - Has an invisible unique id.
  - Has electrical metadata such as tag prefix, tag number, type and partOf.
  - Owns or references one or more shapes on the page.
- `ComponentLabel`
  - A visible label attached to a component instance.
  - Stores relative position, font size, alignment and visibility.
  - Renders like text on the canvas, but is not independent loose text.

Recommended model evolution
- Keep existing project compatibility as much as possible.
- Add new optional structures rather than replacing everything at once.
- Suggested additions:
  - `ProjectComponentInstance`

    - partOfId?: string

  - `ComponentLabel`

Open naming decision
- Current `Component` type name is used for reusable definitions.
- To avoid confusion, consider renaming in code later:
  - Current `Component` -> `ComponentDefinition`
  - New placed entity -> `ComponentInstance`

Risks and implementation notes
- Existing project import is intentionally strict. `componentInstances` is now required for the new format.
- Undo/redo must include component instance metadata or the UI will feel inconsistent.
- The linked label must not become a normal TextShape, or future tag updates and cross-references become fragile.
- Component labels are rendered in CanvasView and in PDF export, so canvas/PDF are aligned for this feature.
- If a component is deleted, componentInstances must be cleaned up.
- If a group/component is ungrouped, decide whether the component instance is deleted or converted to linked shapeIds.
- If user duplicates/copies a component, decide whether the duplicate gets:
  - a new componentId with same derived tag, or
  - a new componentId and suggested next tag number.
- For electrical CAD, duplicate derived tags may be valid for related parts, but accidental duplicates are also possible.
- A future duplicate-tag warning may be useful, but should not block advanced electrical workflows.
- Duplicate-tag checks should compare the derived tag value, not only the prefix.
